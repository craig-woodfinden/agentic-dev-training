# L1.2 — Prompting fundamentals

**Level:** 1 — Novice
**Time:** ~90 minutes
**Pair or solo:** Solo for the exercise; share results with your pair at the end.

## You'll be able to

- Apply five concrete prompt patterns you'll reuse weekly.
- Tell when a prompt needs an example, a format spec, or a role — rather than just rewording it.
- Spot the difference between a system prompt and a user prompt, and when each one matters.

## Read / watch (20 min)

- [Anthropic — Prompt engineering overview](https://docs.claude.com/en/docs/build-with-claude/prompt-engineering/overview) — the full page.
- [Anthropic — System prompts](https://docs.claude.com/en/docs/build-with-claude/prompt-engineering/system-prompts) — skim.
- [Anthropic — Prompt engineering: be clear and direct](https://docs.claude.com/en/docs/build-with-claude/prompt-engineering/be-clear-and-direct) — read in full.

## Do (60 min)

You'll apply five patterns to the same real task — extracting structured info from messy text — and compare results.

**Setup (5 min)**

Find a real piece of messy text from your work: a long Slack thread, a meeting note, a customer email, a JIRA comment. Anonymise it if needed. Paste it into a working file as `INPUT`.

Your job for each pattern: extract a structured summary — `{ "decisions": [...], "owners": [...], "open_questions": [...] }`.

**Pattern 1 — Bare instruction (10 min)**

Send a one-line prompt: `"Summarise the decisions, owners, and open questions from this text: <INPUT>"`. Save the output.

**Pattern 2 — Instruction + format spec (10 min)**

Add an explicit format: `"Return strict JSON with keys decisions (list of strings), owners (list of strings), open_questions (list of strings)."`. Run again.

**Pattern 3 — Add an example (few-shot) (10 min)**

Prepend a short worked example showing one paragraph of input and the exact JSON output you want. Then your real input. Run again.

**Pattern 4 — System prompt + role (10 min)**

Move the standing instructions ("you are an analyst who extracts structured information…") into the `system` parameter. Keep the user message focused on the input. Run again.

**Pattern 5 — Prefill the answer (10 min)**

Use the `assistant` message with a partial start of the JSON (`{"decisions": [`) to force the model into structured output from the first token. (Search the docs for "prefill" if you've not seen this.) Run again.

**Compare (5 min)**

Open all five outputs side by side. Which version produced the cleanest, most parseable result? Which would you trust to drop into a script? Save the winner as `prompt_v1.txt` — that's your first reusable prompt.

## Self-check before your check-in (5 min)

Make sure you can answer these from memory:

- Name the five patterns.
- Which pattern made the biggest difference for your task, and which made surprisingly little?
- When would adding few-shot examples *hurt* a prompt (think: tokens, biasing)?

## Check-in with your engineer lead (~15 min)

Book 15 minutes via Slack DM. Bring:

- The five outputs side by side (in a file or doc).
- Your `prompt_v1.txt` — the winner you saved.
- The original messy `INPUT` you used (anonymised if needed).

The lead will ask you to walk through what you tried for 5–7 minutes, then probe with questions like:

- "Walk me through `prompt_v1.txt`. Which patterns did you combine and why?"
- "Show me a case where pattern X failed and pattern Y succeeded. What was the actual cause?"
- "If you doubled the few-shot examples, what do you predict would happen?"

When the **Done when** bar below is met, the lead signs you off in your `status.md`.

## Done when

You have one prompt saved as a reusable file, and you can name the five patterns from memory.
