# L3.2 -- Production observability for agents

**Level:** 3 -- Proficient
**Time:** ~90 minutes
**Pair or solo:** Solo build; cohort review of dashboards.

## You'll be able to

- Trace an agent run end-to-end (model calls, tool calls, latencies, costs).
- Set up the three alerts every production agent needs.
- Distinguish a model regression from an infra regression by looking at the data.

## Read / watch (20 min)

- [AWS Bedrock -- CloudWatch metrics for Bedrock](https://docs.aws.amazon.com/bedrock/latest/userguide/monitoring-cw.html) -- the primary reference for observability when running agents on Bedrock. AWS CloudWatch is the default monitoring layer for Bedrock deployments.
- Pick one open-source LLM observability tool and read its docs for ten minutes: Langfuse or Helicone are both solid choices that complement CloudWatch with request-level tracing and cost tracking.
- If your team has its own tracing setup, get a tour from a teammate who's used it.

## Do (60 min)

You'll instrument your L2.2 agent with structured tracing and stand up a minimal dashboard.

**Step 1 -- Pick a tool (5 min)**

Use what your team uses. If the team doesn't have one yet and you're on AWS, default to **AWS CloudWatch** -- it's already available with your Bedrock deployment and requires no additional infrastructure. Langfuse self-hosted or OpenTelemetry piped to your existing observability backend are good alternatives if you need richer per-request traces.

**Step 2 -- Instrument every model and tool call (25 min)**

Wrap your agent loop so that every step emits one structured event with at least:

```typescript
// emitted as JSON lines to stdout
{
  trace_id: "...",          // group calls in one run
  step: 3,
  kind: "model_call" | "tool_call",
  model: "anthropic.claude-3-5-sonnet-20241022-v2:0",
  tool: "read_file",        // if tool_call
  input_tokens: 1280,
  output_tokens: 245,
  cache_read_tokens: 1100,
  latency_ms: 1420,
  cost_usd: 0.0034,
  stop_reason: "tool_use",
  error: null,              // or the exception message
}
```

Note: Bedrock model IDs use the format `anthropic.claude-3-5-sonnet-20241022-v2:0` rather than short aliases. Use the full ID in your trace events so you can correlate with CloudWatch metrics.

If you're using an SDK, hook into its existing tracing. If not, a simple wrapper works:

```typescript
async function withTrace<T>(kind: string, fields: Record<string, unknown>, fn: () => Promise<T>): Promise<T> {
  const t0 = Date.now();
  try {
    const result = await fn();
    emit({ kind, latency_ms: Date.now() - t0, error: null, ...fields });
    return result;
  } catch (e) {
    emit({ kind, latency_ms: Date.now() - t0, error: String(e), ...fields });
    throw e;
  }
}

function emit(event: Record<string, unknown>): void {
  process.stdout.write(JSON.stringify(event) + "\n");
}
```

Pipe output to a file: `npx ts-node your_agent.ts | tee traces.jsonl`

**Step 3 -- Build a one-screen dashboard (20 min)**

If you're already on AWS, **CloudWatch** is the natural default: create a CloudWatch dashboard using the Bedrock-emitted metrics (invocation count, latency, throttling) alongside your custom `cost_usd` and error metrics published via `PutMetricData`. Langfuse and Grafana are equally valid if you want a richer per-request view or are instrumenting across multiple providers.

Whichever tool you choose, the dashboard must show:

- Requests per hour (last 24h).
- Median and p95 latency per step.
- Cost per request and total cost today.
- Error rate.
- Tool-use rate (% of runs that called any tool).

**Step 4 -- The three alerts (10 min)**

Define these (even if you wire them up later). Each one should answer a different question:

1. **"Is anything on fire?"** -- Error rate > 5% over 5 minutes.
2. **"Are we hemorrhaging money?"** -- Cost in the last hour > 2x the 7-day rolling average.
3. **"Did quality drop?"** -- Eval pass rate in CI dropped below threshold (this links to L3.4). Or, if you can't run evals live yet: tool-use rate dropped by >30% (often means the model is short-circuiting). In your CI environment, authenticate using `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` (or an IAM role) rather than a provider-specific API key -- Bedrock uses standard AWS credentials.

## Self-check (5 min)

Make sure you can answer these from memory:

- Can you answer "which run cost most yesterday and why?" using only your emitted data?
- Name the three alerts and the threshold you'd set for each.
- How would you distinguish a model regression from an infra regression looking only at the dashboard?

## Done when

You can run a query that answers "which run cost the most yesterday, and why?" using the data your agent emits. You can also name the three alerts you'd want before going to production.
