# L2.2 -- Tool use and function calling

**Level:** 2 -- Practitioner
**Time:** ~90 minutes
**Pair or solo:** Solo build; share tool schemas with the cohort at the end.

## You'll be able to

- Design a tool schema the model will actually use correctly.
- Implement the tool-call loop (model → tool → result → model) end-to-end.
- Spot and prevent the three classic failure modes: hallucinated arguments, infinite loops, schema drift.

## Do (60 min)

Build a tiny agent that can answer questions about your team's repos using two tools you define.

**Setup (5 min)**

**bash / zsh (Mac, Linux)**
```bash
npm install
export ANTHROPIC_API_KEY="..."
```

**PowerShell (Windows)**
```powershell
npm install
$env:ANTHROPIC_API_KEY = "..."
```

Pick one repo on disk you'll let the agent search.

**Define two tools (15 min)**

Write JSON-schema definitions for these two tools. Aim for descriptions that someone unfamiliar with your codebase could understand:

```typescript
import Anthropic from "@anthropic-ai/sdk";

const TOOLS: Anthropic.Messages.Tool[] = [
  {
    name: "list_files",
    description: "List files in the given directory of the repo. Use this to discover what code exists before reading it. Returns up to 100 file paths.",
    input_schema: {
      type: "object" as const,
      properties: {
        path: { type: "string", description: "Path relative to repo root. Empty string means repo root." }
      },
      required: ["path"],
    },
  },
  {
    name: "read_file",
    description: "Read the contents of a file in the repo. Use after list_files to inspect code. Files larger than 50KB are truncated.",
    input_schema: {
      type: "object" as const,
      properties: {
        path: { type: "string", description: "Path relative to repo root." }
      },
      required: ["path"],
    },
  },
];
```

**Implement the loop (30 min)**

```typescript
import * as fs from "fs";
import * as path from "path";

const REPO = path.resolve("/path/to/your/repo");
const client = new Anthropic();

function listFiles(filePath: string) {
  const p = path.resolve(REPO, filePath);
  if (!p.startsWith(REPO)) return { error: "path escapes repo" }; // don't skip this check
  return { files: fs.readdirSync(p).slice(0, 100).map(f => path.relative(REPO, path.join(p, f))) };
}

function readFile(filePath: string) {
  const p = path.resolve(REPO, filePath);
  if (!p.startsWith(REPO)) return { error: "path escapes repo" };
  try { return { content: fs.readFileSync(p, "utf-8").slice(0, 50_000) }; }
  catch (e) { return { error: String(e) }; }
}

const messages: Anthropic.Messages.MessageParam[] = [
  { role: "user", content: "Where is the rate-limit handling in this repo?" }
];

(async () => {
  for (let step = 0; step < 10; step++) {  // hard cap -- see "what goes wrong" below
    const resp = await client.messages.create({
      model: "claude-sonnet-4-6", max_tokens: 1024, tools: TOOLS, messages,
    });
    messages.push({ role: "assistant", content: resp.content });
    if (resp.stop_reason !== "tool_use") {
      const text = resp.content.find(b => b.type === "text");
      console.log("\n--- final ---\n", text?.type === "text" ? text.text : "");
      break;
    }
    const toolResults: Anthropic.Messages.ToolResultBlockParam[] = [];
    for (const block of resp.content) {
      if (block.type === "tool_use") {
        const input = block.input as { path: string };
        const result = block.name === "list_files" ? listFiles(input.path)
                     : block.name === "read_file"  ? readFile(input.path)
                     : { error: `unknown tool ${block.name}` };
        toolResults.push({ type: "tool_result", tool_use_id: block.id, content: JSON.stringify(result) });
      }
    }
    messages.push({ role: "user", content: toolResults });
  }
})();
```

Run it: `npx ts-node repo_agent.ts /path/to/repo "your question"`

Run it. Watch what the model picks.

**Break it on purpose (10 min)**

1. Remove the `range(10)` cap. Ask a vague question. Does the model loop forever, or does it stop? Put the cap back.
2. Drop the security check (`startswith(REPO)`). Ask it to read `/etc/passwd`. (Then put the check back. This is *exactly* why tool scoping matters.)
3. Make a tool description deliberately misleading. Does the model still use the tool correctly, or does it follow your description?

## Self-check before your check-in (5 min)

Make sure you can answer these from memory:

- Name the three failure modes (loop-no-stop, hallucinated args, schema drift / misleading description).
- Which of the three did you actually hit?
- Read your tool descriptions aloud -- would a stranger understand when to use them?

## Check-in with your engineer lead (~15 min)

Book 15 minutes with your engineer lead. Bring:

- Your `repo_agent.ts` (with the safety checks restored -- don't ship the broken version).
- A real question your agent answered correctly + a transcript.
- A short note on the worst failure mode you hit and how you fixed (or scoped around) it.

The lead will ask you to walk through what you built for 5-7 minutes, then probe with questions like:

- "Walk me through the loop. What stops it?"
- "Show me one tool description. What's the strictest version you can write?"
- "Tell me about the worst failure mode you hit. Was it the model, or your description?"

When the **Done when** bar below is met, the lead signs you off in your `status.md`.

## Done when

You have a working agent that uses tools to answer a real question about your repo, and you can name the three failure modes from memory.
