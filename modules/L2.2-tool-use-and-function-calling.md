# L2.2 -- Tool use and function calling

**Level:** 2 -- Practitioner
**Time:** ~90 minutes
**Pair or solo:** Solo build; share tool schemas with the cohort at the end.

## You'll be able to

- Design a tool schema the model will actually use correctly.
- Implement the tool-call loop (model → tool → result → model) end-to-end.
- Spot and prevent the three classic failure modes: hallucinated arguments, infinite loops, schema drift.

## Read / watch (15 min)

- [OpenAI -- Function calling](https://platform.openai.com/docs/guides/function-calling) -- full page; focus on the loop diagram and how tool results are sent back.
- [GitHub Models docs](https://docs.github.com/en/github-models/prototyping-with-ai-models) -- same endpoint, same format as the Chat Completions API you used in L2.1.

## Do (60 min)

Build a tiny agent that can answer questions about your team's repos using two tools you define.

**Setup (5 min)**

**bash / zsh (Mac, Linux)**
```bash
export GITHUB_TOKEN="ghp_..."   # your GitHub PAT
```

**PowerShell (Windows)**
```powershell
$env:GITHUB_TOKEN = "ghp_..."
```

Pick one repo on disk you'll let the agent search.

**Define two tools (15 min)**

Write the function definitions for these two tools. Aim for descriptions that someone unfamiliar with your codebase could understand:

```typescript
import OpenAI from "openai";

const TOOLS: OpenAI.Chat.ChatCompletionTool[] = [
  {
    type: "function",
    function: {
      name: "list_files",
      description: "List files in the given directory of the repo. Use this to discover what code exists before reading it. Returns up to 100 file paths.",
      parameters: {
        type: "object",
        properties: {
          path: { type: "string", description: "Path relative to repo root. Empty string means repo root." }
        },
        required: ["path"],
      },
    },
  },
  {
    type: "function",
    function: {
      name: "read_file",
      description: "Read the contents of a file in the repo. Use after list_files to inspect code. Files larger than 50KB are truncated.",
      parameters: {
        type: "object",
        properties: {
          path: { type: "string", description: "Path relative to repo root." }
        },
        required: ["path"],
      },
    },
  },
];
```

**Implement the loop (30 min)**

```typescript
import * as fs from "fs";
import * as path from "path";

const REPO = path.resolve("/path/to/your/repo");
const client = new OpenAI({
  baseURL: "https://models.inference.ai.azure.com",
  apiKey: process.env.GITHUB_TOKEN,
});
const MODEL = "gpt-4o";   // or "claude-3-5-sonnet" -- both available on GitHub Models

function listFiles(filePath: string) {
  const p = path.resolve(REPO, filePath);
  if (!p.startsWith(REPO)) return { error: "path escapes repo" }; // don't skip this check
  return { files: fs.readdirSync(p).slice(0, 100).map(f => path.relative(REPO, path.join(p, f))) };
}

function readFile(filePath: string) {
  const p = path.resolve(REPO, filePath);
  if (!p.startsWith(REPO)) return { error: "path escapes repo" };
  try { return { content: fs.readFileSync(p, "utf-8").slice(0, 50_000) }; }
  catch (e) { return { error: String(e) }; }
}

const messages: OpenAI.Chat.ChatCompletionMessageParam[] = [
  { role: "user", content: "Where is the rate-limit handling in this repo?" }
];

(async () => {
  for (let step = 0; step < 10; step++) {  // hard cap -- see "what goes wrong" below
    const resp = await client.chat.completions.create({
      model: MODEL, max_tokens: 1024, tools: TOOLS, messages,
    });
    const message = resp.choices[0].message;
    messages.push(message);

    if (!message.tool_calls || message.tool_calls.length === 0) {
      console.log("\n--- final ---\n", message.content ?? "");
      break;
    }

    for (const toolCall of message.tool_calls) {
      const input = JSON.parse(toolCall.function.arguments) as { path: string };
      const result = toolCall.function.name === "list_files" ? listFiles(input.path)
                   : toolCall.function.name === "read_file"  ? readFile(input.path)
                   : { error: `unknown tool ${toolCall.function.name}` };
      messages.push({
        role: "tool",
        tool_call_id: toolCall.id,
        content: JSON.stringify(result),
      });
    }
  }
})();
```

Run it: `npx ts-node repo_agent.ts`

Watch what the model picks.

**Break it on purpose (10 min)**

1. Remove the `step < 10` cap. Ask a vague question. Does the model loop forever, or does it stop? Put the cap back.
2. Drop the security check (`p.startsWith(REPO)`). Ask it to read `/etc/passwd`. (Then put the check back. This is *exactly* why tool scoping matters.)
3. Make a tool description deliberately misleading. Does the model still use the tool correctly, or does it follow your description?

## Self-check (5 min)

Make sure you can answer these from memory:

- Name the three failure modes (loop-no-stop, hallucinated args, schema drift / misleading description).
- Which of the three did you actually hit?
- Read your tool descriptions aloud -- would a stranger understand when to use them?

## Done when

You have a working agent that uses tools to answer a real question about your repo, and you can name the three failure modes from memory.
