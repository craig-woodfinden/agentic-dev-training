# L3.4 — Evals at scale

**Level:** 3 — Proficient
**Time:** ~90 minutes
**Pair or solo:** Solo; share judge prompts with the cohort.

## You'll be able to

- Write an LLM-as-judge that grades open-ended output without lying to you.
- Run evals in CI so prompt changes are gated, not just hoped about.
- Use sampled production traces as fresh eval inputs.

## Read / watch (20 min)

- [Anthropic — LLM-as-judge / model-graded evals](https://docs.claude.com/en/docs/test-and-evaluate/develop-tests) — read on grading methods.
- [Anthropic — Reducing hallucinations / consistency techniques](https://docs.claude.com/en/docs/test-and-evaluate/strengthen-guardrails/reduce-hallucinations) — skim.
- Read your team's existing eval setup if there is one.

## Do (60 min)

You'll upgrade the eval set from L2.3 into something you'd actually trust to gate a deploy.

**Step 1 — Add LLM-as-judge for the open-ended cases (25 min)**

Substring matching can only get you so far. For each case where the expected answer is qualitative, write a judge prompt. A solid pattern:

```python
JUDGE_PROMPT = """You are grading model outputs. Be strict.

Question to the model:
{input}

Rubric for a good answer:
{rubric}

Model output:
{output}

Reply with strict JSON only:
{{"score": <0, 1, or 2>, "reason": "<one sentence>"}}

0 = wrong or misses the point. 1 = partially right. 2 = correct and well-formed.
"""

def judge(input_, rubric, output):
    resp = client.messages.create(
        model="claude-sonnet-4-6",   # not Haiku — Haiku is too lenient as a judge
        max_tokens=200,
        messages=[{"role": "user", "content": JUDGE_PROMPT.format(input=input_, rubric=rubric, output=output)}],
    )
    return json.loads(resp.content[0].text)
```

Important: **calibrate your judge first.** Grade ten cases manually, then run the judge over the same ten. If the judge disagrees with you more than once or twice, tighten the rubric. A bad judge is worse than no judge.

**Step 2 — Wire it into CI (20 min)**

Add a GitHub Action (or your equivalent) that runs the eval on every PR touching the prompt:

```yaml
# .github/workflows/evals.yml
on:
  pull_request:
    paths: ['prompts/**', 'src/agent/**']
jobs:
  evals:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: '3.12' }
      - run: pip install -r requirements.txt
      - run: python -m evals.run --threshold 0.85
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
```

The runner exits non-zero if pass rate falls below the threshold. A PR that drops eval scores can't be merged without an explicit override.

**Step 3 — Pull production traces back into the eval set (15 min)**

Open your dashboard from L3.2. Find five real production inputs from the last week that you don't already have in the eval set — ideally a mix of routine and weird. Anonymise. Add them as new cases. Re-run.

This is the loop that matters: production teaches the eval set, the eval set protects production.

## Self-check before your check-in (5 min)

Make sure you can answer these from memory:

- How did you calibrate your judge prompt against your own manual scores?
- What threshold did you set for CI, and why that number?
- Which production traces taught you something new that you've now codified as cases?

## Check-in with your engineer lead (~15 min)

Book 15 minutes via Slack DM. Bring:

- Your judge prompt.
- The CI workflow YAML (and a link to a passing/failing CI run if you have one).
- The five (or more) new cases you sourced from production traces.

The lead will ask you to walk through the judge for 5–7 minutes, then probe with questions like:

- "Show me a case where you manually graded 0 — does your judge agree?"
- "Try to game your judge live. Where does it crack?"
- "What's the cadence for backfilling from production? Who owns it?"

When the **Done when** bar below is met, the lead signs you off in your `status.md`.

## Done when

You have an eval set with at least one judge-graded case, a CI workflow that blocks PRs on regressions, and at least five new cases sourced from real production traces.
