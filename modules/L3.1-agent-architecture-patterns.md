# L3.1 — Agent architecture patterns

**Level:** 3 — Proficient
**Time:** ~90 minutes
**Pair or solo:** Pair recommended — pattern selection sharpens with debate.

## You'll be able to

- Pick between ReAct, planner-executor, router, and sub-agent patterns based on the problem.
- Draw a working diagram of an agent before writing the code.
- Spot which pattern is hiding in someone else's agent.

## Read / watch (20 min)

- [Anthropic — Agent SDK overview](https://docs.claude.com/en/docs/claude-code/agent-sdk) — read the architecture section.
- [Anthropic — Building effective agents](https://www.anthropic.com/research/building-effective-agents) — the patterns section is the canonical reference. Read in full.
- Look at one real agent in your codebase (or any of Anthropic's example agents) and ask: which pattern is this?

## Do (60 min)

You won't build a full agent — you'll design three.

**Step 1 — Three problems (5 min)**

Pick three problems from your team's backlog or imagination. They should be different in shape. Examples:

- A. Triage incoming support tickets and either reply, file a bug, or escalate.
- B. Take a vague product brief and produce a structured eng spec.
- C. Investigate a flaky test by running it, reading logs, and proposing a fix.

**Step 2 — Match patterns to problems (30 min)**

For each problem, sketch one architecture. Write each as half a page (or draw it on paper). Cover:

- Which pattern: ReAct (think-act loop) / planner-executor (plan first, then run) / router (classify then dispatch) / sub-agent (decompose to specialists).
- What tools the agent has.
- Where memory or retrieval lives, if anywhere.
- What stops the loop. (Step cap. Confidence threshold. User confirmation.)
- One failure mode you expect, and how the design handles it.

Rough heuristics — keep them in mind as you decide:

- **Router** when the work splits cleanly into a handful of known paths. Cheap, fast, easy to reason about.
- **ReAct** when the task is exploratory and the next step depends on the previous one.
- **Planner-executor** when the work has expensive steps and a wrong turn is costly.
- **Sub-agents** when subtasks are different enough that one prompt can't carry them — and you'd happily run them in parallel.

**Step 3 — Defend one (15 min)**

Pick the design you're least sure about. Argue against it. Write three reasons it might be the wrong choice and what you'd switch to. Then commit to a final answer.

**Step 4 — Spot the pattern (10 min)**

Pull up an agent that already exists — in your repo, in [Anthropic's cookbook](https://github.com/anthropics/anthropic-cookbook), or in the Agent SDK examples. Read it for five minutes. Which pattern is it using? Is there a clearer alternative?

## Self-check before your check-in (5 min)

Make sure you can answer these from memory:

- Name the four patterns and one sentence on when to pick each.
- For the design you argued against — what would you switch to, and what would trigger the switch?
- The existing agent you read: which pattern is it using?

## Check-in with your engineer lead (~15 min)

Book 15 minutes via Slack DM. Bring:

- Your three architecture sketches.
- The one you argued against, plus your written argument.
- A pointer to the existing agent you analysed.

The lead will ask you to walk through one sketch for 5–7 minutes (pick the most ambitious), then probe with questions like:

- "Why this pattern over the other three?"
- "What's the simplest version that still works? Did you over-design it?"
- "On the existing agent: would you change its pattern? On what evidence?"

When the **Done when** bar below is met, the lead signs you off in your `status.md`.

## Done when

You have three written or drawn architecture sketches, one of which you've argued against and then committed to. You can name the four patterns and recognise them in someone else's code.
