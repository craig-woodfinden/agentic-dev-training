# L1.3 -- Daily GitHub Copilot workflow

**Level:** 1 -- Novice
**Time:** ~75 minutes
**Pair or solo:** Solo, then share your comment pattern and one failure note with your pair.

## You'll be able to

- Drive GitHub Copilot inline completions and Chat end-to-end for a small, real task.
- Write a comment that reliably produces useful inline completions.
- Use Copilot Chat with `@workspace` and `#file` references to give the model meaningful context.

## Read / watch (15 min)

- [GitHub Copilot -- Getting started in VS Code](https://docs.github.com/en/copilot/getting-started-with-github-copilot?tool=vscode) -- read the quickstart.
- [GitHub Copilot -- Using Copilot Chat in your IDE](https://docs.github.com/en/copilot/github-copilot-chat/using-github-copilot-chat-in-your-ide) -- read the section on context variables (`@workspace`, `#file`).
- [GitHub Copilot -- Best practices](https://docs.github.com/en/copilot/using-github-copilot/best-practices-for-using-github-copilot) -- skim.

## Do (50 min)

You'll verify your setup, practice inline completions, use Chat to understand and reshape code, then ship a small real change.

**Setup (10 min)**

1. Open VS Code in a repo you actually work in.
2. Confirm the Copilot extension is installed and signed in: check the Copilot icon in the status bar (bottom right). It should show a filled icon, not a disabled one.
3. Open a file in that repo. Type a comment and wait two seconds. If a greyed-out suggestion appears, Copilot inline completions are working. Press `Tab` to accept it.
4. Open the Chat panel (`Ctrl+Alt+I` / `Cmd+Alt+I`). Send a test message: `"What does this repo do?"` using `@workspace`. If you get a coherent answer, Chat is working.

If either is not working, fix it before continuing. The rest of this module will not work without both.

**Inline completions (15 min)**

Pick a function you need to write -- something real, not toy code. Write only a comment describing what the function should do. Do not write the function body.

Wait for a Copilot suggestion. If the suggestion is wrong or vague, rewrite the comment to be more specific and try again. Iterate on the comment until the completion is useful.

A comment that produces a good completion typically:
- Names the input and output types explicitly.
- States the edge case the function must handle.
- Gives the function a name with a clear verb.

When you have a comment pattern that works, copy it into a scratch note. That is the pattern to save and reuse.

**Copilot Chat (15 min)**

Find a function in the repo that is genuinely confusing -- something you'd have to think about to explain to a colleague.

1. In the Chat panel, ask: `"Explain what this function does and why it's structured this way. #file:<filename>"` (replace `<filename>` with the actual filename). Read the explanation. Is it accurate?
2. Ask a follow-up: `"Refactor this function so the early-exit conditions are easier to read. Show me the diff, don't apply it yet."` Review what it proposes. Does it preserve the behavior?
3. If the proposal looks safe, ask it to apply the change: `"Apply that refactor."` Then read the changed code.

Note one thing Copilot got wrong or missed in the explanation or refactor. Note it for your self-assessment.

**Drive a small real change (10 min)**

Pick a small real chore in this repo: a rename, a missing test, a docstring sweep, a small bug. Use Copilot inline completions and Chat together to complete it. Some prompts to try:

- `"Write a docstring for this function. #file:<filename>"`
- `"Show me the smallest possible diff that would rename X to Y across this file."`
- `"Write a unit test for this function. Follow the test patterns already in #file:<test-filename>."`

Commit the work in a branch. Do not merge yet.

## Self-check (5 min)

Make sure you can answer these from memory:

- What is the comment pattern you saved that produced the best inline completion?
- Name one thing Copilot got wrong on your small change, and how did you catch it?
- When would you use inline completions rather than Chat, and when would you use Chat rather than inline?

## Done when

You have:
- A comment pattern saved that you would use again in real work
- A branch with a Copilot-assisted commit that you'd be willing to put up for review
- Notes on one thing Copilot got wrong and how you caught it
