# L2.3 — Building your first eval set

**Level:** 2 — Practitioner
**Time:** ~90 minutes
**Pair or solo:** Pair recommended — eval rubric design is much sharper with two opinions.

## You'll be able to

- Author a 10-case eval set with a clear pass/fail rubric.
- Run an eval and read its results without lying to yourself.
- Catch at least one real regression before it ships.

## Read / watch (15 min)

- [Anthropic — Evaluating your prompts](https://docs.claude.com/en/docs/build-with-claude/develop-tests) — full page.
- Skim one open-source eval framework's README to see the shape (Inspect AI, promptfoo, or DeepEval — pick one).

## Do (60 min)

You'll build a focused eval set for one prompt you actually use (from L1.2 or a real prompt from your work).

**Step 1 — Pick the unit under test (5 min)**

Pick *one* prompt. Don't try to eval a whole feature today. The narrower the better.

**Step 2 — Author 10 cases (25 min)**

Open a CSV or JSONL file. Each row has:

- `input` — the input to your prompt (a real one from logs is ideal).
- `expected` — what good output looks like. Can be a regex, a JSON shape, or a free-text rubric.
- `category` — one of: `happy_path`, `edge_case`, `failure_mode`, `regression`.

Aim for roughly: 4 happy paths, 3 edge cases, 2 failure modes, 1 regression you've actually seen.

**Examples of categories you want to cover:**

- Happy path: the obvious case the prompt was built for.
- Edge case: empty input, very long input, weird Unicode, mixed languages.
- Failure mode: input that *should* refuse or escalate (out-of-scope, injection-shaped).
- Regression: a real bug you (or the team) hit before. The eval exists so it doesn't come back.

**Step 3 — Write the runner (20 min)**

```python
import csv, json, anthropic

client = anthropic.Anthropic()

def run_prompt(input_text):
    resp = client.messages.create(
        model="claude-haiku-4-5",
        max_tokens=512,
        messages=[{"role": "user", "content": YOUR_PROMPT.format(input=input_text)}],
    )
    return resp.content[0].text

def grade(output, expected, category):
    # Start simple: substring or regex match.
    # Upgrade to LLM-as-judge in L3.4 — not yet.
    return expected.lower() in output.lower()

with open("evals.csv") as f, open("results.csv", "w", newline="") as out:
    reader = csv.DictReader(f)
    writer = csv.writer(out)
    writer.writerow(["category", "input", "passed", "output"])
    passed = total = 0
    for row in reader:
        output = run_prompt(row["input"])
        ok = grade(output, row["expected"], row["category"])
        writer.writerow([row["category"], row["input"], ok, output])
        passed += ok; total += 1
    print(f"{passed}/{total} passed")
```

**Step 4 — Catch a regression (10 min)**

Now break your prompt on purpose. Remove a critical instruction. Re-run the eval. You should see specific cases fail.

Put the instruction back. Re-run. They pass again.

This is the whole point: the eval set lets you change a prompt with confidence.

## Self-check before your check-in (5 min)

Make sure you can answer these from memory:

- Name the four categories and how many cases you have in each.
- Describe the regression you broke and which specific case caught it.
- Where would substring matching fool you? When would you upgrade to a judge?

## Check-in with your engineer lead (~15 min)

Book 15 minutes with your engineer lead. Bring:

- Your `evals.csv` (at least 10 cases, all four categories).
- Your `results.csv` from the latest run.
- Notes on the regression you broke and the case(s) that caught it.

The lead will ask you to walk through what you built for 5–7 minutes, then probe with questions like:

- "Walk me through one case from each category. Why this one?"
- "Show me the regression — re-run it live if you can."
- "Where would your `grade()` fool you? What would you change first?"

When the **Done when** bar below is met, the lead signs you off in your `status.md`.

## Done when

You have:
- A CSV/JSONL of at least 10 cases covering all four categories
- A runner script that produces a pass/fail count
- A demonstrated regression you broke and fixed using the eval
