# L2.1 -- Calling an LLM API

**Level:** 2 -- Practitioner
**Time:** ~90 minutes
**Pair or solo:** Solo build, then code review with your pair.

## You'll be able to

- Make Anthropic Messages API calls with system prompts, multi-turn conversations, and streaming.
- Handle the three errors that will actually happen in production: rate limits, network timeouts, context too long.
- Explain the difference between a one-shot call and a streamed response, and when to use each.

## Read / watch (15 min)

- [Anthropic -- Getting started](https://docs.anthropic.com/en/docs/get-started) -- read through the setup and first request.
- [Anthropic -- Messages API reference](https://docs.anthropic.com/en/api/messages) -- read the request schema in full, skim the response section.
- [Anthropic -- Streaming](https://docs.anthropic.com/en/api/messages-streaming) -- full page. You'll implement this in the Do section.

## Do (60 min)

Build a tiny multi-turn chat CLI that streams responses. Real, runnable, ~80 lines.

**Setup (5 min)**

```bash
npm install @anthropic-ai/sdk
```

**bash / zsh (Mac, Linux)**
```bash
export ANTHROPIC_API_KEY="sk-ant-..."
```

**PowerShell (Windows)**
```powershell
$env:ANTHROPIC_API_KEY = "sk-ant-..."
```

**Build the chat (40 min)**

Save as `mini_chat.ts`. Your version should:

1. Hold a `messages` list across turns (multi-turn).
2. Stream the response token-by-token using the streaming API.
3. Print neatly to stdout.
4. On any 429 rate-limit error, sleep with exponential backoff and retry up to 3 times.

Skeleton to get you moving:

```typescript
import Anthropic from "@anthropic-ai/sdk";
import * as readline from "readline";

const client = new Anthropic({ apiKey: process.env.ANTHROPIC_API_KEY });

const MODEL = "claude-haiku-4-5-20251001";
const SYSTEM = "You are a concise senior engineer.";
const messages: Anthropic.MessageParam[] = [];
const rl = readline.createInterface({ input: process.stdin, output: process.stdout });
const ask = (q: string) => new Promise<string>((resolve) => rl.question(q, resolve));

(async () => {
  while (true) {
    const userInput = (await ask("you> ")).trim();
    if (!userInput) continue;
    messages.push({ role: "user", content: userInput });

    process.stdout.write("assistant> ");
    let full = "";

    const stream = client.messages.stream({
      model: MODEL,
      max_tokens: 1024,
      system: SYSTEM,
      messages,
    });

    for await (const text of stream.textStream) {
      process.stdout.write(text);
      full += text;
    }

    const finalMsg = await stream.finalMessage();
    console.log();
    console.log(`(tokens: ${finalMsg.usage.input_tokens} in / ${finalMsg.usage.output_tokens} out)`);

    messages.push({ role: "assistant", content: full });
  }
})();
```

Run it: `npx ts-node mini_chat.ts`

Make it your own. Add a `/reset` command that clears the message history. Add retry logic for 429 rate-limit errors with exponential backoff. Try swapping to `claude-sonnet-4-6` and compare response quality and latency.

**Break it on purpose (10 min)**

Force at least two failure modes and observe what the SDK throws:

1. Set `max_tokens: 1` and ask for a long answer. What does the response look like?
2. Spam ten requests in a tight loop. Does the SDK retry? Do you?
3. Pass a message list that grows very long. What happens to response quality and latency?

Make sure your retry logic actually handles each one.

## Self-check (5 min)

Make sure you can answer these from memory:

- Which three errors does your retry loop actually handle?
- What is the difference between `input_tokens` and `output_tokens`, and which costs more?
- What happens to response quality when the message list grows longer than the model's context window?

## Done when

You can show a working multi-turn streaming chat that survives a forced rate-limit error without crashing. You can show the token count per turn and explain how the retry backoff works.
