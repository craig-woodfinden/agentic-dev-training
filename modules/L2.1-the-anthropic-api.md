# L2.1 — The Anthropic API

**Level:** 2 — Practitioner
**Time:** ~90 minutes
**Pair or solo:** Solo build, then code review with your pair.

## You'll be able to

- Make Messages API calls with system prompts, multi-turn conversations, and streaming.
- Handle the three errors that will actually happen in production: rate limits, network timeouts, content too long.
- Read the SDK source for the language you actually use, well enough to debug.

## Read / watch (15 min)

- [Anthropic — Messages API reference](https://docs.claude.com/en/api/messages) — read the request schema in full; skim the response section.
- [Anthropic — Streaming](https://docs.claude.com/en/api/streaming) — read the events list.
- [Anthropic — Errors](https://docs.claude.com/en/api/errors) — full page.

## Do (60 min)

Build a tiny multi-turn chat CLI that streams responses. Real, runnable, ~80 lines.

**Setup (5 min)**

```bash
pip install anthropic rich
export ANTHROPIC_API_KEY="..."
```

**Build the chat (40 min)**

Save as `mini_chat.py`. Your version should:

1. Hold a `messages` list across turns (multi-turn).
2. Accept a `--system` flag to set the system prompt.
3. Stream the response token-by-token using `client.messages.stream(...)`.
4. Print neatly (use `rich` if you like).
5. On any `anthropic.APIError`, log the error type, sleep with exponential backoff, retry up to 3 times.

Skeleton to get you moving:

```python
import sys, time
import anthropic

client = anthropic.Anthropic()
system = "You are a concise senior engineer."
messages = []

while True:
    try:
        user = input("you> ")
    except EOFError:
        break
    messages.append({"role": "user", "content": user})

    with client.messages.stream(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        system=system,
        messages=messages,
    ) as stream:
        print("claude> ", end="", flush=True)
        full = ""
        for text in stream.text_stream:
            print(text, end="", flush=True)
            full += text
        print()
    messages.append({"role": "assistant", "content": full})
```

Make it your own. Add a `/reset` command. Add token counting per turn. Print the cost after each response.

**Break it on purpose (10 min)**

Force at least two failure modes and observe what the SDK throws:

1. Set `max_tokens=1` and ask for a long answer. What does the response look like?
2. Spam ten requests in a tight loop. Does the SDK retry? Do you?
3. Pass a message list that grows past the model's context window. What error?

Make sure your retry logic actually handles each one.

**Read the SDK (5 min)**

Open the [anthropic Python SDK on GitHub](https://github.com/anthropics/anthropic-sdk-python) (or the SDK for your language). Find the file that defines `messages.stream`. Read it until you understand one new thing. Take a screenshot of the line that surprised you.

## Self-check before your check-in (5 min)

Make sure you can answer these from memory:

- Which three errors does your retry loop actually handle?
- What happens when you grow the message list past the model's context window?
- Which line in the SDK source surprised you, and why?

## Check-in with your engineer lead (~15 min)

Book 15 minutes with your engineer lead. Bring:

- Your `mini_chat.py`, ready to run live.
- The screenshot of the SDK line that surprised you (or a link to it).
- Notes on how you tested rate-limit handling.

The lead will ask you to walk through what you did for 5–7 minutes, then probe with questions like:

- "Run `mini_chat.py` live. Force a rate-limit. Walk me through what the retry loop does."
- "Show me how you count cost per turn. How would you log it for a real product?"
- "Point me at the SDK source line and explain it."

When the **Done when** bar below is met, the lead signs you off in your `status.md`.
