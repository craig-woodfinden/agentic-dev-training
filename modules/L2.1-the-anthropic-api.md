# L2.1 -- Calling an LLM API

**Level:** 2 -- Practitioner
**Time:** ~90 minutes
**Pair or solo:** Solo build, then code review with your pair.

## You'll be able to

- Make Chat Completions API calls with system prompts, multi-turn conversations, and streaming.
- Handle the three errors that will actually happen in production: rate limits, network timeouts, context too long.
- Use the GitHub Models endpoint with the same GitHub token you already have for Copilot.

## Read / watch (15 min)

- [GitHub Models quickstart](https://docs.github.com/en/github-models/prototyping-with-ai-models) -- read through the setup and first request.
- [OpenAI Chat Completions API reference](https://platform.openai.com/docs/api-reference/chat) -- GitHub Models uses this exact format; read the request schema in full, skim the response section.
- [GitHub Models rate limits](https://docs.github.com/en/github-models/prototyping-with-ai-models#rate-limits) -- full page.

## Do (60 min)

Build a tiny multi-turn chat CLI that streams responses. Real, runnable, ~80 lines.

**Setup (5 min)**

```bash
npm install openai
```

**bash / zsh (Mac, Linux)**
```bash
export GITHUB_TOKEN="ghp_..."   # your GitHub PAT -- same one you use for git push
```

**PowerShell (Windows)**
```powershell
$env:GITHUB_TOKEN = "ghp_..."
```

**Build the chat (40 min)**

Save as `mini_chat.ts`. Your version should:

1. Hold a `messages` list across turns (multi-turn).
2. Stream the response token-by-token using the `stream: true` option.
3. Print neatly to stdout.
4. On any 429 rate-limit error, sleep with exponential backoff and retry up to 3 times.

Skeleton to get you moving:

```typescript
import OpenAI from "openai";
import * as readline from "readline";

const client = new OpenAI({
  baseURL: "https://models.inference.ai.azure.com",
  apiKey: process.env.GITHUB_TOKEN,
});

const MODEL = "gpt-4o";   // or "claude-3-5-sonnet" -- both available on GitHub Models
const system = "You are a concise senior engineer.";
const messages: OpenAI.Chat.ChatCompletionMessageParam[] = [];
const rl = readline.createInterface({ input: process.stdin, output: process.stdout });
const ask = (q: string) => new Promise<string>((resolve) => rl.question(q, resolve));

(async () => {
  while (true) {
    const user = (await ask("you> ")).trim();
    if (!user) continue;
    messages.push({ role: "user", content: user });

    const stream = await client.chat.completions.create({
      model: MODEL,
      max_tokens: 1024,
      messages: [{ role: "system", content: system }, ...messages],
      stream: true,
    });

    process.stdout.write("assistant> ");
    let full = "";
    for await (const chunk of stream) {
      const delta = chunk.choices[0]?.delta?.content ?? "";
      process.stdout.write(delta);
      full += delta;
    }
    console.log();
    messages.push({ role: "assistant", content: full });
  }
})();
```

Run it: `npx ts-node mini_chat.ts`

Make it your own. Add a `/reset` command that clears the message history. Print a token count estimate per turn (GitHub Models returns usage in the final chunk). Add retry logic for 429 rate-limit errors with exponential backoff.

**Break it on purpose (10 min)**

Force at least two failure modes and observe what the SDK throws:

1. Set `max_tokens: 1` and ask for a long answer. What does the response look like?
2. Spam ten requests in a tight loop. Does the SDK retry? Do you?
3. Pass a message list that grows very long. What happens to response quality and latency?

Make sure your retry logic actually handles each one.

Note: GitHub Models is for prototyping. L2.4 covers AWS Bedrock for production deployments.

## Self-check (5 min)

Make sure you can answer these from memory:

- Which three errors does your retry loop actually handle?
- What happens when the message list grows very long?
- How does the GitHub Models endpoint relate to the OpenAI API format?

## Done when

You can show a working multi-turn streaming chat that survives a forced rate-limit error without crashing. You can also show the token count per turn and explain how the retry backoff works.
