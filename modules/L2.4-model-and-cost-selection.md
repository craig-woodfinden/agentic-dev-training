# L2.4 -- AWS Bedrock for production

**Level:** 2 -- Practitioner
**Time:** ~75 minutes
**Pair or solo:** Solo build; cohort comparison at the end.

## You'll be able to

- Call a Claude model via AWS Bedrock and compare it to the same call via the direct Anthropic API.
- Explain when to use the Anthropic API directly vs Bedrock in a production architecture.
- Estimate Bedrock costs for a feature before it ships.

## Read / watch (15 min)

- [AWS Bedrock -- What is Amazon Bedrock?](https://docs.aws.amazon.com/bedrock/latest/userguide/what-is-bedrock.html) -- overview.
- [AWS Bedrock -- Converse API](https://docs.aws.amazon.com/bedrock/latest/userguide/conversation-inference.html) -- the standardised inference interface.
- [AWS Bedrock -- Pricing](https://aws.amazon.com/bedrock/pricing/) -- full page.
- [AWS Bedrock -- Supported models](https://docs.aws.amazon.com/bedrock/latest/userguide/models-supported.html) -- skim the model list.

## Do (50 min)

You'll make your first Bedrock call, run your L2.3 eval set against it, and write a short decision memo comparing Bedrock to the direct Anthropic API.

**Step 1 -- Set up Bedrock credentials (5 min)**

```bash
# Install AWS SDK
npm install @aws-sdk/client-bedrock-runtime

# Configure AWS credentials (ask your manager for the team IAM role)
aws configure   # or set AWS_ACCESS_KEY_ID / AWS_SECRET_ACCESS_KEY / AWS_DEFAULT_REGION
```

Check access:

```bash
aws bedrock list-foundation-models --region us-east-1 | head -30
```

**Step 2 -- Make your first Bedrock call (15 min)**

Use the Converse API (standardised across all Bedrock models):

```typescript
import { BedrockRuntimeClient, ConverseCommand } from "@aws-sdk/client-bedrock-runtime";

const client = new BedrockRuntimeClient({ region: "us-east-1" });

const command = new ConverseCommand({
  modelId: "anthropic.claude-3-5-sonnet-20241022-v2:0",
  messages: [{ role: "user", content: [{ text: "Write a four-line poem about debugging at 2am." }] }],
  inferenceConfig: { maxTokens: 200, temperature: 0.7 },
});

const response = await client.send(command);
const text = response.output?.message?.content?.[0];
if (text?.text) console.log(text.text);
console.log("Input tokens:", response.usage?.inputTokens);
console.log("Output tokens:", response.usage?.outputTokens);
```

**Step 3 -- Run your L2.3 eval set against both providers (20 min)**

Adapt your eval runner from L2.3 to accept a `--provider` flag:

- `--provider anthropic` -- Anthropic SDK with `ANTHROPIC_API_KEY` (what you built in L2.1)
- `--provider bedrock` -- BedrockRuntimeClient with AWS credentials

Run the same 10 cases on both. Record for each provider:

- Pass rate.
- Total tokens (input + output).
- Total cost (compute from the Bedrock pricing page vs Anthropic pricing page).
- Median per-call latency.

Note: the Anthropic API and Bedrock both serve Claude models, but latency, cost structure, and compliance posture differ. The direct API is simpler to set up; Bedrock keeps everything inside AWS.

**Step 4 -- Write a one-page decision memo (10 min)**

Save as `provider_decision.md`. Answer:

- For this eval set: which provider would you use in production, and why?
- What's the monthly cost estimate at 10k calls/day on Bedrock?
- What would tip you from the Anthropic API to Bedrock? (Data residency? AWS compliance requirements? IAM integration?)
- What region would you deploy Bedrock in, and why?

A good memo is a screen of text, not five pages. Make it the thing you'd send your manager when they ask "why are we paying for Bedrock?".

## Self-check (5 min)

Make sure you can answer these from memory:

- The cost per 1,000 Bedrock calls for the model you used -- with a number.
- One case where the two providers differed in latency.
- When you would prefer the direct Anthropic API over Bedrock, and vice versa.

## Done when

You have a working Bedrock call, a cost comparison against the direct Anthropic API, and a one-page memo defending which provider to use for a real feature.
